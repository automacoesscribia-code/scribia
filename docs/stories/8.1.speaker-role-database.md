## Story

**As a** system,
**I want** the speaker role added to the database schema with proper user account linking,
**so that** speakers can have real accounts, log in, and be associated with their lectures.

## Context

Atualmente o sistema tem 3 roles (`super_admin`, `organizer`, `participant`) e a tabela `speakers` é um registro simples sem vínculo com `auth.users` ou `user_profiles`. Para o fluxo completo do palestrante funcionar, precisamos:

1. Adicionar `speaker` ao enum `user_role`
2. Vincular `speakers` a `user_profiles` via `user_id`
3. Permitir `speaker` na tabela `invitations`
4. Atualizar trigger `handle_new_user()` para lidar com convites de speaker
5. Criar RLS policies para speakers acessarem apenas suas proprias palestras

**Regra de negocio:** O speaker e vinculado ao evento/organizador no momento do convite. Ao aceitar, sua conta e criada e linkada ao registro na tabela `speakers`.

## Acceptance Criteria

1. Enum `user_role` inclui valor `speaker`
2. Tabela `speakers` tem coluna `user_id UUID REFERENCES user_profiles(id)` (nullable — null ate aceitar convite)
3. Tabela `invitations` aceita `role = 'speaker'` no CHECK constraint
4. Tabela `invitations` tem coluna `speaker_id UUID REFERENCES speakers(id)` (nullable — so para convites de speaker)
5. Trigger `handle_new_user()` atualizado: quando invitation.role = 'speaker', seta `speakers.user_id = NEW.id`
6. RLS em `lectures`: speaker pode SELECT em lectures onde `speaker_id` aponta para speaker com `user_id = auth.uid()`
7. RLS em `speakers`: speaker pode SELECT/UPDATE seu proprio registro
8. RLS em `events`: speaker pode SELECT eventos que tem palestras vinculadas a ele
9. Middleware aceita role `speaker` e redireciona para rota adequada
10. Helper function `is_speaker()` retorna BOOLEAN

## Dev Notes

**Migration SQL:**
```sql
-- Adicionar speaker ao enum (requer transacao separada em Postgres)
ALTER TYPE public.user_role ADD VALUE IF NOT EXISTS 'speaker';

-- Vincular speakers a user accounts
ALTER TABLE public.speakers ADD COLUMN user_id UUID REFERENCES public.user_profiles(id);
CREATE INDEX idx_speakers_user_id ON public.speakers(user_id);

-- Permitir speaker em invitations
ALTER TABLE public.invitations DROP CONSTRAINT IF EXISTS invitations_role_check;
ALTER TABLE public.invitations ADD CONSTRAINT invitations_role_check
  CHECK (role IN ('organizer', 'participant', 'speaker'));
ALTER TABLE public.invitations ADD COLUMN speaker_id UUID REFERENCES public.speakers(id);
```

**Trigger update:**
```sql
-- Dentro de handle_new_user(), apos a logica existente:
IF v_invitation.role = 'speaker' AND v_invitation.speaker_id IS NOT NULL THEN
  UPDATE public.speakers SET user_id = NEW.id WHERE id = v_invitation.speaker_id;
END IF;
```
