## Story

**As a** speaker (or super_admin),
**I want** my recorded audio to be saved locally on my computer first, and then deploy it to the cloud when I choose,
**so that** I always have a local copy of my recordings and control when to upload.

## Context

Atualmente o app grava audio em chunks WAV na pasta temp e o upload e feito logo apos parar a gravacao via botao "Enviar para nuvem". O fluxo precisa mudar para:

1. **Gravacao → salva LOCAL automaticamente** (pasta permanente, nao temp)
2. Audio permanece salvo no computador do usuario
3. **Usuario decide quando fazer "Deploy"** (upload para Supabase Storage)
4. Botao "Deploy" claramente separado do fluxo de gravacao
5. Indicador visual de status: "Salvo localmente" → "Enviando..." → "Publicado"

**Regra:** Super admin tambem pode gravar e fazer deploy (alem de speakers).

**Pasta local:** `{app_data_dir}/recordings/{eventId}/{lectureId}/` — pasta de dados do app (AppData no Windows, Application Support no macOS).

## Acceptance Criteria

1. Ao parar gravacao, audio (chunks WAV) e salvo na pasta de dados do app: `{app_data_dir}/recordings/{eventId}/{lectureId}/`
2. Pasta permanente sobrevive reinicializacao do app e do sistema
3. Tela pos-gravacao mostra: "Audio salvo em [caminho]" com botao para abrir pasta
4. Botao "Deploy" (purple, destaque) disponivel SOMENTE apos gravacao salva localmente
5. Deploy faz upload dos chunks para Supabase Storage (`audio-files/{eventId}/{lectureId}/`)
6. Durante deploy: progress bar com porcentagem + status "Enviando chunk X de Y"
7. Apos deploy bem-sucedido: status muda para "Publicado" (badge verde) + timestamp
8. Se deploy falhar: mensagem de erro + botao "Tentar novamente"
9. Na lista de palestras (Story 8.3), cards mostram status correto:
   - "Agendada" (sem audio) — chip azul
   - "Salvo localmente" (tem chunks locais, nao fez deploy) — chip amarelo
   - "Publicado" (deploy feito) — chip verde
10. Super admin pode fazer deploy de qualquer palestra
11. Ao voltar para lista de palestras, o botao Deploy permanece disponivel para audios nao publicados
12. Gravacao anterior nao e sobrescrita — novo recording cria subpasta com timestamp

## Dev Notes

**Pasta permanente (Rust):**
```rust
use tauri::api::path::document_dir;

#[tauri::command]
fn get_recordings_dir(event_name: String, lecture_title: String) -> Result<String, String> {
    let docs = document_dir().ok_or("Cannot find Documents directory")?;
    let dir = docs.join("ScribIA").join(&event_name).join(&lecture_title)
        .join(chrono::Local::now().format("%Y%m%d_%H%M%S").to_string());
    std::fs::create_dir_all(&dir).map_err(|e| e.to_string())?;
    Ok(dir.to_string_lossy().to_string())
}

#[tauri::command]
fn open_folder(path: String) -> Result<(), String> {
    open::that(&path).map_err(|e| e.to_string())
}
```

**Status logic:**
```typescript
function getLectureStatus(lecture: Lecture, localChunks: LocalChunkInfo): LectureDisplayStatus {
  if (lecture.status === 'completed' || lecture.status === 'processing') return 'published'
  if (localChunks.exists && localChunks.chunk_count > 0) return 'saved-locally'
  return 'scheduled'
}
```

**Deploy flow:**
```typescript
async function handleDeploy() {
  setDeployStatus('uploading')
  const result = await invoke('upload_chunks', {
    eventId, lectureId, outputDir: recordingPath,
    supabaseUrl, supabaseKey
  })
  if (result.success) {
    // Atualizar status da lecture no banco
    await supabase.from('lectures').update({ status: 'processing' }).eq('id', lectureId)
    setDeployStatus('published')
  } else {
    setDeployStatus('error')
  }
}
```
